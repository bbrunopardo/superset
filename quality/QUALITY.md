# Quality

Quality is defined as observable behavior + engineering invariants. It must be gradable, not "feels good".

## Priority flows

These are the two flows we are actively trying to validate end to end. Everything else is a known gap (see bottom).

### Flow 1 — Ontology proposal lifecycle

**User behavior**
1. An agent makes one or more iterations on the ontology.
2. Iterations are bundled into a proposal artifact.
3. A reviewer opens the proposal and can read enough to judge it without leaving the surface.
4. The reviewer validates the proposal (mechanical check passes) and accepts it.
5. On acceptance, the changes apply to the canonical ontology.

**Acceptance criteria (to be sharpened in a plan)**
- [ ] An agent can produce a proposal containing a structured diff of ontology changes.
- [ ] The proposal surfaces, at minimum: the diff, the agent's justification, downstream impact, and any validation evidence.
- [ ] A reviewer can accept or reject the proposal in one explicit action.
- [ ] Accepted proposals apply atomically — either all changes land, or none.
- [ ] Rejected proposals are preserved with the reviewer's reason.
- [ ] All proposal state transitions are auditable (who, when, why).

**Evidence of success**
- A reproducible end-to-end walkthrough captured in `runbooks/` once the flow exists.
- A test or fixture per acceptance criterion.

### Flow 2 — Context-graph visualization for multi-step agent runs

**User behavior**
1. An agent runs a task that takes multiple steps.
2. Each step consults specific parts of the context graph and applies specific governance rules.
3. After (or during) the run, the user can view a visual that shows: each step in order, and which parts of the context graph were involved in each step.

**Acceptance criteria (to be sharpened in a plan)**
- [ ] Each step emits a structured record naming the context-graph nodes/edges it consulted and the governance rules it applied.
- [ ] The records can be reconstructed into a step-by-step trace without losing ordering.
- [ ] The visual shows: the sequence of steps, the graph subset touched per step, and the governance rules in play.
- [ ] The visual is reachable from the surface where the agent runs.
- [ ] A reviewer can answer "what informed this decision?" by clicking a step.

**Evidence of success**
- One reference run captured as a fixture in `state/` or `docs/`, plus a screenshot of the corresponding visual.
- The user-supplied reference material is linked from `docs/decisions/` once received.

## Engineering invariants

Always true regardless of feature work:

- **Type safety** — avoid `any` unless justified at the site.
- **Layering** — apps depend on packages; packages never depend on apps. `packages/ui` is presentation-only.
- **Boundaries** — inputs from network or filesystem are parsed at the boundary, not trusted as raw JSON.
- **Schema ownership** — `packages/db` owns the schema. Migrations are auto-generated by Drizzle. Never hand-edit `packages/db/drizzle/`.
- **Next.js 16** — request interception is `proxy.ts`, never `middleware.ts`.
- **Co-location** — new code follows the structure rules in `AGENTS.md` (`Component/Component.tsx` + `index.ts`; tests next to source).
- **Single source of MCP truth** — `.mcp.json` is canonical; per-tool configs mirror it.
- **Plans, not chat** — anything spanning sessions lives in `plans/`, not in chat history.
- **State updates reflect reality** — `state/features.json` and `state/progress.md` are only updated to mirror verified progress.

## Operability

- Every priority flow gets a runbook entry under `runbooks/` describing how to exercise it locally.
- Failures should produce a recoverable artifact (log line, screenshot, fixture) that an agent can attach to a follow-up plan.

## Security and compliance

- Secrets stay in `.env`; never commit `.env*` files.
- Production database is off-limits unless the user explicitly asks; even then, confirm before touching it.
- Package-age policy (7-day minimum release, `ignore-scripts=true` for npm) is enforced — do not bypass.

- Performance, reliability, and security sections need concrete numbers once the priority flows are running.
